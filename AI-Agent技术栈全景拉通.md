# AI Agent 技术栈全景拉通

> 目标：拉通 AI/LLM/Agent/RAG 的底层逻辑，让你能独立闭环一个 AI Agent 项目。

---

## 一、先揪头发：你站在什么位置看 AI？

AI Agent 不是凭空冒出来的，它是一条技术演进链上的产物：

```
规则系统 → 机器学习(ML) → 深度学习(DL) → 大语言模型(LLM) → AI Agent
 (if/else)   (学模式)      (神经网络)      (理解+生成)      (自主行动)
```

**底层逻辑**：每一代的跃迁，本质上是把"人要做的事"逐步交给机器。Agent 是这条链上最新的一环——它不是被动回答问题，而是**主动规划→调用工具→执行→反思→再执行**的自主系统。

---

## 二、核心概念颗粒度拆解

### 1. LLM（大语言模型）— 一切的地基

**是什么**：一个巨大的神经网络，用海量文本训练出来的"语言理解+生成"模型。GPT-4、Claude、DeepSeek 都是 LLM。

**底层逻辑**：LLM 本质上做一件事——**给定一段文本，预测下一个 token（词元）**。但在海量数据和参数加持下，"预测下一个词"涌现出了推理、翻译、编程、创意写作等能力。

关键概念：

| 概念 | 一句话解释 | 抓手在哪 |
|------|-----------|---------|
| **Token** | 文本的最小单位，一个汉字≈1-2 token，一个英文单词≈1-3 token | 计费、上下文窗口都按 token 算 |
| **Context Window** | 模型一次能"看到"的最大文本量 | Agent 记忆管理、RAG 召回策略都受它制约 |
| **Temperature** | 控制输出的随机性，0=确定，1=狂野 | 需要精确时调低，需要创意时调高 |
| **Hallucination（幻觉）** | 模型一本正经地胡说八道 | 这是 RAG 存在的核心原因之一 |
| **System Prompt** | 给模型设定角色和规则的"元指令" | Agent 的行为边界靠它定义 |

---

### 2. Prompt Engineering — 你和模型的接口

**是什么**：设计输入文本（prompt）来精准控制 LLM 输出的方法论。

**底层逻辑**：LLM 是"下一个词预测器"，它不会主动思考"你想要什么"——它只会根据 prompt 的模式延续下去。Prompt 的质量直接决定输出的质量。

核心技巧（三板斧）：

- **Few-shot**：给几个示例，让模型照猫画虎
- **Chain-of-Thought（CoT）**：要求模型"一步一步思考"，推理题必备
- **结构化输出**：要求返回 JSON/YAML，让下游代码能解析

**颗粒度对齐**：Prompt Engineering 是 Agent 的"指令层"。Agent 的规划、反思、工具调用，底层全是 prompt 在驱动。

---

### 3. RAG（检索增强生成）— 给 LLM 装上外挂知识库

**是什么**：在 LLM 生成回答之前，先从外部知识库检索相关信息，拼到 prompt 里一起喂给模型。

**为什么需要 RAG？** LLM 有三大硬伤：

| 硬伤 | 问题 | RAG 怎么解决 |
|------|------|-------------|
| **知识截止** | 训练数据有截止日期 | 从实时文档/数据库检索最新信息 |
| **幻觉** | 不知道就说不知道，但偏偏瞎编 | 检索事实性文档，约束模型基于文档回答 |
| **领域盲区** | 通用模型不懂你的内部业务 | 从企业内部知识库检索，让模型"懂业务" |

**RAG 的完整链路（核心闭环）**：

```
文档 → 切块(chunking) → 向量化(embedding) → 存入向量数据库
                                                    ↓
用户提问 → 向量化 → 相似度检索 → 召回Top-K文档 → 拼入Prompt → LLM生成 → 回答
```

关键子概念：

| 概念 | 一句话 | 颗粒度细节 |
|------|:------:|-----------|
| **Embedding** | 把文本转成高维向量，语义相近的文本向量距离近 | 这是 RAG 的"心脏" |
| **Vector DB** | 专门存向量+做相似度检索的数据库（Milvus、Pinecone、ChromaDB、pgvector） | 是整个检索链路的性能瓶颈 |
| **Chunking** | 把长文档切成小块，块太大召回不精、太小丢上下文 | RAG 质量的第一道关卡 |
| **Re-ranking** | 初检索后再精排一次，把最相关的放前面 | 检索质量不够就加这步 |
| **Hybrid Search** | 向量检索 + 关键词检索组合使用 | 精确查询（名称、编号）关键词更准 |

### 4. AI Agent — 这代技术的顶层设计

**是什么**：能自主感知环境、制定计划、调用工具、执行行动、反思调整的 AI 系统。

**Agent ≠ LLM**。LLM 是大脑，Agent 是整个身体。大脑负责思考，身体负责感知和行动。

**Agent 的四层架构**：

```
┌─────────────────────────────────────────┐
│              记忆 (Memory)                │
│   短期记忆 = 对话历史                      │
│   长期记忆 = 向量库/数据库持久化            │
├─────────────────────────────────────────┤
│              规划 (Planning)              │
│   任务拆解 + 执行路径决策                  │
│   关键技术: ReAct / Plan-and-Execute      │
├─────────────────────────────────────────┤
│              工具 (Tools)                 │
│   搜索API / 代码执行 / 数据库查询 / 文件读写  │
│   关键技术: Function Calling / MCP       │
├─────────────────────────────────────────┤
│              行动 (Action)                │
│   调用工具 → 观察结果 → 反思 → 调整 → 再行动 │
└─────────────────────────────────────────┘
```

**Agent 的核心循环**：

```
感知(Observation) → 思考(Thought) → 行动(Action) → 观察结果 → 反思 → 再行动
                    ↑_______________________________________________↓
                                    自主循环
```

关键框架：

| 框架 | 定位 | 一句话 |
|------|------|--------|
| **ReAct** | 范式 | Reason + Act：边想边做，想了做，做了再想 |
| **LangChain** | 开发框架 | Agent 开发的"脚手架"，链式调用+工具集成 |
| **LangGraph** | 状态机框架 | 用图结构编排 Agent 流程，比 LangChain 更灵活 |
| **AutoGPT** | 早期实验 | 第一个出圈的自主 Agent，"给定目标自己拆解执行" |
| **MCP** | 协议标准 | Anthropic 提出的 Agent-工具通信协议，类似 USB-C 统一接口 |

---

### 5. MCP（Model Context Protocol）— Agent 的 USB-C 接口

**是什么**：一个开放协议，标准化了 AI 模型和外部工具/数据源之间的通信方式。

**底层逻辑**：没有 MCP 之前，每个 Agent 框架自己定义工具调用格式——形成 M×N 的集成问题（M个框架 × N个工具 = 爆炸）。MCP 统一了接口，变成 M+N。**这是 Agent 工程化的基础设施。**

```
Agent ──MCP协议──→ MCP Server (数据库)
      ──MCP协议──→ MCP Server (文件系统)
      ──MCP协议──→ MCP Server (API网关)
      ──MCP协议──→ MCP Server (搜索)
```

## 三、概念之间的闭环关系

```
                    ┌──────────────────────────────┐
                    │         AI Agent              │
                    │  ┌────────┐  ┌────────────┐  │
                    │  │ 规划器  │  │  记忆系统   │  │
                    │  └───┬────┘  └─────┬──────┘  │
                    │      │              │         │
                    │  ┌───┴──────────────┴───┐     │
                    │  │    LLM (大脑)         │     │
                    │  │  受 Prompt 控制       │     │
                    │  └───┬──────────────────┘     │
                    └──────┼────────────────────────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         ┌────┴────┐  ┌────┴────┐  ┌───┴──────┐
         │   RAG   │  │  MCP    │  │ 其他工具  │
         │ 检索知识 │  │ 标准化  │  │  API调用  │
         │   库     │  │ 工具接口 │  │           │
         └────┬────┘  └─────────┘  └──────────┘
              │
    ┌─────────┼─────────┐
    │         │         │
┌───┴──┐ ┌───┴───┐ ┌───┴───┐
│向量DB│ │Embedding│ │文档处理│
│      │ │  模型   │ │(Chunk) │
└──────┘ └───────┘ └───────┘
```

**一句话总结**：**Agent 是顶层应用形态，LLM 是推理引擎，RAG 是知识注入手段，MCP 是工具连接标准，Prompt 是指令接口。**

## 四、动手路线图（从 0 到能独立交付）

新人最容易犯的错误是光学不练。以下是**实践闭环路径**，按顺序走：

### 第一阶段：玩转 LLM（1-2 天）

- 注册 API（OpenAI / Claude / DeepSeek）
- 写一个最简单的 Python 脚本调用 API
- 实验 system prompt 对输出的影响
- 实验 temperature 参数

### 第二阶段：搭一个最简 RAG（2-3 天）

- 选一个文档，切成 chunks
- 调 embedding API 转向量
- 存入 ChromaDB（轻量向量库，纯 Python，零配置安装）
- 实现流程：提问 → 向量化 → 检索 → 拼 prompt → 生成回答
- 对比有/无 RAG 的差异（同一个问题，感受幻觉 vs 有依据的区别）

### 第三阶段：做一个最小 Agent（3-5 天）

- 给 LLM 配一个搜索工具（比如 SerpAPI 或者简单的计算器）
- 实现 ReAct 循环：思考 → 调工具 → 观察 → 再思考
- 让 Agent 能自主决定"这个问题需要搜索"还是"我直接能回答"

### 第四阶段：工程化（持续）

- 引入 LangChain/LangGraph 管理流程
- 接 MCP Server
- 加上记忆系统（对话历史 + 长期记忆向量库）
- 评估系统（RAGAS 评估检索质量）

## 五、冰山下面的东西

你没问但做项目时一定会遇到的：

| 概念 | 一句话 | 什么时候会用到 |
|------|--------|---------------|
| **Fine-tuning（微调）** | 在基座模型上用特定领域数据再训练 | RAG 效果不够好、需要模型学会特定风格或格式 |
| **Function Calling** | LLM 输出结构化 JSON 触发函数执行（Agent 调用工具的底层机制） | 做 Agent 第一天就会遇到 |
| **Multi-Agent** | 多个 Agent 分工协作，各自有专长 | 复杂项目需要拆解成多个独立子任务 |
| **Guardrails（护栏）** | 输入输出安全过滤，防注入、防越狱、防敏感内容 | 上生产环境必须加 |
| **Evaluation（评估）** | 系统化评估 Agent/RAG 效果的框架 | 不是"感觉好用"就行，要有数据闭环 |
